Context: choosing between React Router and TanStack Router

        - Vite

        - Tailwind + shadcn/ui

        - Data-heavy UI (10k+ items, filtering, charts)

        Project goals:
        - Strong performance story (Lighthouse metrics, virtualization, memoization)
        - Clean, production-like architecture (Senior-level, not overengineered)
        - Fast development (2–3 weeks total)
        - Portfolio-quality code, not a reusable library

        I need you to compare:
        - React Router
        - TanStack Router

        Specifically for THIS project (not in general).

        Focus on:
        1. Developer experience in a small-to-medium SPA
        2. Type safety and whether it actually adds value here
        3. Integration with TanStack Query (is there real synergy or just marketing)
        4. Boilerplate / cognitive overhead
        5. Performance implications (if any, realistically)
        6. Learning curve vs ROI for a portfolio project
        7. Risk of overengineering

        Then:
        - Give a clear recommendation (pick ONE)
        - Explain WHY the other option is worse in THIS context
        - Mention what would change your recommendation (edge cases)

        Avoid generic statements like “both are good” — I want a decisive answer with trade-offs.
Result:  Why TanStack Router is worse for this project

          1. The killer features don't pay off at 4 routes.
          TanStack Router's marquee value is typed search params + typed loaders + full link type-safety. That payoff scales with route count and URL-state complexity. With 4 routes and filters that mostly live
          in TanStack Query keys (per your CLAUDE.md), you'd be paying setup cost for type guarantees you don't actually need.

          2. The "TanStack Query synergy" is mostly marketing for your shape of app.
          The real synergy — router-level loaders that hydrate the QueryClient before render — matters most for SSR, waterfall avoidance on slow networks, and apps where you want data ready before the component
          mounts. You're a client-only SPA with MSW mocks and a dashboard pattern where useQuery inside components is idiomatic and fine. You'd wire up loader: () => queryClient.ensureQueryData(...) and gain
          ~nothing visible.

          3. Boilerplate / cognitive overhead is real.
          TanStack Router wants a Vite plugin, generated route tree, file-based conventions, and createRootRoute/createFileRoute wrappers. For 2–3 weeks of build time, that's days of router learning that doesn't
           show up in your portfolio story.

          4. Your performance story is elsewhere.
          The headline perf demo is 10k-row virtualization, lazy Recharts, and bundle splitting. Router preloading is rounding error next to those. Picking TanStack Router does not strengthen the Lighthouse
          before/after — it just changes which router renders the route.

          5. Portfolio signal is neutral.
          Reviewers don't grade router choice. They grade whether your architecture is clean and your perf claims hold up. React Router v7 is the boring, defensible, "I picked the right tool for the size"
          answer.

          ---
          What would flip the recommendation

          - 20+ routes or deeply nested layouts — typed <Link to> and route trees start saving real time.
          - Filters lifted to URL as the source of truth (e.g., /transactions?from=...&to=...&cat=...&q=... shareable/back-button-correct) — TanStack Router's typed/validated search params become the obvious
          win, and Zustand-for-filters disappears.
          - SSR or static prerendering in scope — TanStack Router's loader model matures here.
          - You wanted the router itself to be a portfolio talking point (e.g., applying for jobs at companies betting on the TanStack stack). Then learning it is the value.

          None of those apply to your spec as written.

          ---
          Concrete next step

          Keep react-router-dom v7. Use a <BrowserRouter> with the four routes from SPEC §8, lazy-load the route components (React.lazy + Suspense) so Recharts and the transactions virtualization stay out of the
           initial bundle. Filters stay in Zustand or query keys per your existing convention. Don't introduce route loaders — let useQuery inside pages do its job.
